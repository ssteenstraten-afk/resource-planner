Weekly backup of the entire allocation state, created every Sunday at 23:00.

ColumnTypeDescription
snapshot_week / snapshot_jaarintegerThe week the snapshot was taken

jaar / week / urenintegerSnapshot values

Key Design Choices

Assignment via bezetting records: When a planner assigns a consultant to a project, 0-hour records are created for all weeks in the project period. This means the consultant immediately sees the project in their overview without having to fill in hours first.

ISO week numbers: All week calculations use ISO 8601 (week starts on Monday). The date-fns library handles edge cases around year transitions.

Unique constraint: The bezetting table has a unique constraint on (consultant_id, project_id, jaar, week), which makes upserts safe.

Functionality

Use Cases

Consultant

1
Sign in
Consultant

The consultant signs in via the login page using either a password or a magic link sent by email.

  • Navigate to the app URL
  • Enter work email address
  • Choose: sign in with password or request a magic link
  • After successful authentication, redirected to My weekly overview
2
View own weekly allocation
Consultant

The consultant sees their own allocation for the next 12 weeks across all assigned projects.

  • Open My weekly overview
  • See a grid: projects as rows, weeks as columns
  • Bottom rows show: allocated hours, contract hours, available hours, percentage
  • Color coding: red (<60%), orange (60–79%), green (≥80%), purple (>100%)
3
Edit own hours
Consultant

The consultant fills in or adjusts their hours per project per week.

  • Click on a cell in the weekly overview
  • Enter the number of hours
  • Changed cells are highlighted with an orange border
  • Click "Save changes" to save all changes at once
4
Receive weekly email
Automated

Every Monday at 07:00, each active consultant receives a personalised email with their 12-week allocation overview and a magic link to log straight in.

Planner

5
View capacity dashboard
Planner

The planner sees a heatmap of all consultants across 12 weeks: percentages and colour coding per cell.

  • Open Dashboard
  • Filter by level or specific consultant (optional)
  • Navigate forward/backward in time with the week buttons
  • Click a cell to see which projects and hours are behind the percentage
6
View consultant detail
Planner

The planner clicks on a consultant name in the dashboard and sees the full weekly breakdown with all projects and hours for that consultant.

7
Create project
Planner

The planner creates a new project with name, client, start/end date and status.

  • Go to Projects
  • Click "+ New project"
  • Fill in name, client (optional), start/end date, status
  • Click "Create project"
8
Assign consultant to project
Planner

The planner assigns a consultant to a project. Zero-hour records are created for all weeks in the project period so the consultant immediately sees the project.

  • Go to Projects
  • Click "Consultants (x)" on the relevant project
  • Select a consultant from the dropdown
  • Click "Assign"
9
Remove consultant from project
Planner

The planner removes a consultant assignment. All bezetting records for that combination are deleted.

  • Open the project's consultant panel
  • Click the × button next to the consultant's name
10
Close / reopen project
Planner

A completed project can be closed. Closed projects are greyed out but remain visible. System projects (Leave, Bench) cannot be closed.

11
Create user
Planner

The planner creates a new user account. A Supabase auth user is created and a consultant record is inserted simultaneously.

  • Go to Consultants
  • Click "+ New user"
  • Fill in name, email, level, contract hours and role
  • Click "Create user" — the user can now sign in with their email address
12
Deactivate / activate user
Planner

The planner can deactivate a user (e.g. after departure). Inactive users no longer appear in the dashboard and receive no emails, but their data is preserved.

Automated (System)

13
Weekly snapshot
Automated

Every Sunday at 23:00, an Edge Function creates a snapshot of the current allocation state in the bezetting_snapshot table. Idempotent: runs again if the same week already has a snapshot.

14
Weekly email dispatch
Automated

Every Monday at 07:00, an Edge Function sends a personalised HTML email to each active consultant with their 12-week allocation overview. Each email contains a magic link that logs the consultant in directly.

Product roadmap

Backlog

The following features have been discussed and are prioritised for future development. They are not yet implemented in the current version.

1
Exact Online integration — one-way sync
High priority

Vintura uses Exact Online for consultant and project administration. A one-way sync from Exact to the Resource Planner would eliminate duplicate data entry.

Scope: Pull consultants and projects from the Exact Online REST API into Supabase. A "Sync from Exact" button triggers an Edge Function that calls the Exact API and upserts the results.

Technical requirements: Register an app in the Exact developer portal to obtain a Client ID and Client Secret. Implement OAuth 2.0 flow (redirect + token refresh). Store the refresh token as a Supabase secret. Map Exact data structure to the Resource Planner data model.

Estimated effort: 1–2 days.

2
Microsoft Teams — personal notifications via Graph API
Medium priority

In addition to the weekly email, consultants could receive a personalised Teams message with their allocation overview.

Scope: Send a direct message (chat) to each consultant in Microsoft Teams every Monday, containing the same allocation overview as the email.

Technical requirements: Register an Azure AD app with Chat.Create and ChatMessage.Send permissions. Implement OAuth 2.0 with admin consent. Store tokens as Supabase secrets. Map Vintura user emails to Teams user IDs via the Graph API.

Note: Channel-based Incoming Webhooks are not suitable as they are not personalised and visible to all channel members.

Estimated effort: 2–3 days.

3
Verified sender domain for Resend
Required for production

Currently the weekly emails are sent from onboarding@resend.dev because no domain has been verified in Resend yet. For production use, emails should come from a Vintura address.

Scope: Verify a (sub)domain in Resend, for example mail.vintura.nl. Add the DNS records provided by Resend to the vintura.nl DNS configuration. Set the FROM_EMAIL secret in Supabase to noreply@mail.vintura.nl.

Technical requirements: Access to the DNS settings of vintura.nl (via IT administrator or domain registrar). Estimated DNS propagation time: up to 24 hours.

Estimated effort: 30 minutes work + propagation wait.
